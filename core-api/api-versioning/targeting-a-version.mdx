If you haven't already, read [Versioning strategy](/core-api/api-versioning/strategy) for the recommended workflow. This page covers the mechanics — how to pin your organization and how to override per request.

## Why target a specific version

- **Predictability**: every call uses the wire format and behavior of one specific date.
- **Incremental migration**: when a new version ships, you can opt a subset of traffic in without changing your organization-wide pin.
- **Reproducibility**: requests embedded in tests or recorded payloads keep behaving the same way over time.

## Default: the pinned version

All requests use your organization's **pinned version** by default. It starts at the most recent version available at the time your organization was created. Check or change it in the [API Keys](https://pro.nabla.com/developers/api-keys) section of the Console. You can pin to any existing version.

The current version is `2026-04-24`. For the complete list, see the [Changelog](/core-api/changelog).

## Override per request

You can override the pinned version on individual requests in two ways. Pick **header** for normal HTTP traffic and **query parameter** for WebSockets (where browsers can't set custom headers).

### Via header

<Tabs>
<Tab title="cURL">

```bash
curl https://us.api.nabla.com/{endpoint} \
  -H "X-Nabla-Api-Version: 2026-04-24"
```

</Tab>
<Tab title="Node">

```js
await fetch("https://us.api.nabla.com/v1/core/server/generate-note", {
  method: "POST",
  headers: {
    authorization: `Bearer ${access_token}`,
    "content-type": "application/json",
    "X-Nabla-Api-Version": "2026-04-24",
  },
  body: JSON.stringify({ ... }),
});
```

</Tab>
<Tab title="Python">

```python
requests.post(
    "https://us.api.nabla.com/v1/core/server/generate-note",
    headers={
        "authorization": f"Bearer {access_token}",
        "X-Nabla-Api-Version": "2026-04-24",
    },
    json={...},
)
```

</Tab>
</Tabs>

### Via query parameter

```bash
curl "https://us.api.nabla.com/{endpoint}?nabla-api-version=2026-04-24"
```

For WebSocket endpoints:

```js
new WebSocket("wss://us.api.nabla.com/v1/core/user/transcribe-ws?nabla-api-version=2026-04-24");
```

## Header or query — pick one

If both are present, the header wins. In practice, pick one strategy per integration:

- Use the **header** in backend code.
- Use the **query parameter** in browser WebSocket code, where setting headers isn't possible.
