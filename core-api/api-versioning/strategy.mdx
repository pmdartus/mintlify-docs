The Core API uses **date-based versioning** (e.g., `2026-04-24`). Every backwards-incompatible change ships under a new dated version; backwards-compatible improvements (model upgrades, bug fixes, new optional fields) roll out across all live versions automatically.

## The three concepts

| Concept | Where it lives | Purpose |
|---|---|---|
| **Pinned version** | Your organization (API Admin Console) | The default version applied to any request that doesn't specify one. |
| **Per-request override** | `X-Nabla-Api-Version` header or `nabla-api-version` query parameter | Targets a specific version for one call — useful during incremental migrations. |
| **Changelog** | This site | The catalogue of every version and what changed. |

## Recommended workflow

1. **Pin your organization** to a known version when you first integrate. This stops surprise upgrades.
2. **Watch the [Changelog](/core-api/api-versioning/changelog)** for new versions. Each entry calls out breaking vs additive changes.
3. **When you decide to upgrade**, use the header override (`X-Nabla-Api-Version: 2026-04-24`) on a subset of traffic first — staging, then 1% of production, then full rollout.
4. **Once all clients run on the new version**, change the pinned version in the API Admin Console to match. The override remains supported, but you can drop it from request paths that no longer need it.
5. **Drop overrides** in code only after the pinned version matches; this keeps the diff small if you need to roll back.

## Deprecation policy

When a version is deprecated, Nabla announces it in the changelog with a sunset date — typically several months out. Until the sunset date the version continues to accept requests; after the sunset date, requests targeting it return `404`. You'll always have ample notice; pre-2024-10-01 versions still work today.
