Nabla Connect delivers every server-to-server callback (note exports, patient instructions exports, and future event types) to a **single endpoint** that you configure in the [Nabla Connect Admin](https://app.nabla.com/admin/nabla-connect). All callbacks share the envelope, headers, and response protocol described here. The per-type payload shapes are documented on their own pages — see [`NOTE_EXPORT`](/connect/reference/callbacks/note-export) and [`PATIENT_INSTRUCTIONS_EXPORT`](/connect/reference/callbacks/patient-instructions-export).

For the narrative version with end-to-end verification snippets, see [Handle callbacks](/connect/guides/handle-callbacks).

## Single-endpoint model

You register **one** HTTPS URL in the Admin. Nabla `POST`s every callback to that URL. Your handler dispatches on the envelope `type` field. Adding support for a new callback type later doesn't require registering a new endpoint — only updating your dispatcher.

## Request headers

<ResponseField name="x-nabla-connect-timestamp" type="string (ISO-8601)" required>
  Timestamp of when Nabla generated the request. Reject (`401`) if older than 60 seconds — replay protection.
</ResponseField>

<ResponseField name="x-nabla-connect-signature" type="string" required>
  One or more HMAC-SHA256 hex signatures of `timestamp + raw_body`, comma-separated. Multiple signatures may appear during a key-rotation window — accept if **any** of them matches.
</ResponseField>

The signature is computed as `HMAC-SHA256(signature_secret, timestamp + raw_request_body)`. `signature_secret` is retrieved from the [Nabla Connect Admin](https://app.nabla.com/admin/nabla-connect).

## Verification recipe

On every incoming request, your endpoint must:

1. Read the **raw** request body before JSON parsing. The HMAC covers the exact bytes Nabla sent.
2. Extract `x-nabla-connect-timestamp` and `x-nabla-connect-signature`.
3. Reject (`401`) if the timestamp is missing or older than **60 seconds**.
4. Compute `HMAC-SHA256(signature_secret, timestamp + raw_body)` as a hex string.
5. Split the `x-nabla-connect-signature` header on `,` and compare against each received signature using a **constant-time** compare (`crypto.timingSafeEqual` in Node, `hmac.compare_digest` in Python). Reject with `401` if none match.
6. Check that `request_uuid` hasn't already been processed. If it has, respond `200` with the same body — Nabla may retry. See [Idempotency](/connect/best-practices/idempotency).

<Warning>
Never compare HMAC signatures with `==` or `===`. Naïve string comparison leaks timing information that lets an attacker forge a valid signature byte-by-byte. Use the platform's constant-time compare.
</Warning>

## Envelope

All callbacks use the same base envelope. The `type` field determines the rest of the payload.

<ResponseField name="request_uuid" type="string" required>
  Unique identifier for this callback delivery. Use it as your idempotency key and echo it back on the `200` response.
</ResponseField>

<ResponseField name="type" type="string" required>
  The callback type. Dispatch on this to parse `data`. Treat unknown values as `400`. Currently one of `NOTE_EXPORT` or `PATIENT_INSTRUCTIONS_EXPORT`.
</ResponseField>

<ResponseField name="data" type="object" required>
  Payload whose shape depends on `type`. See the per-type pages.
</ResponseField>

```json Example envelope
{
  "request_uuid": "4f1a8e16-2c0e-4e63-9c0e-7ac6f6e7c1f0",
  "type": "NOTE_EXPORT",
  "data": { /* type-specific */ }
}
```

## Response protocol

Your endpoint must respond with one of the following status codes.

| Status | When to use | Body |
|---|---|---|
| `200` | Request recognized and processed (or already processed — idempotent). | `{ "request_uuid": "<same uuid>" }` |
| `400` | Unrecognized `type` value. | Optional `{ "error": "..." }`. |
| `401` | Signature verification failed (missing/old timestamp, no matching signature). | Optional `{ "error": "..." }`. |

For every `200` response, echo back the same `request_uuid` you received:

```json
{ "request_uuid": "4f1a8e16-2c0e-4e63-9c0e-7ac6f6e7c1f0" }
```

<Warning>
Nabla awaits your response before showing a success message to the provider in the Nabla app. Slow or failed responses surface an error in the UI. Keep the synchronous path fast — verify, persist a minimal record, and acknowledge. Defer EHR writes and other heavy work to a background queue. See [Callback latency](/connect/best-practices/callback-latency).
</Warning>
