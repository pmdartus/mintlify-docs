---
title: "Upsert user"
description: "Create or update a Nabla Connect provider user, optionally setting specialty and speech locales."
api: "POST /v1/connect/server/users"
---

Create or update a provider user without opening an encounter. Use this endpoint when you want a provider's specialty and dictation locales set **before** the first encounter is launched. For lazy provisioning at launch time, see [`POST /encounters`](/connect/reference/server-api/create-encounter).

<Note>
Available from API version `2026-03-23`. Send the `X-Nabla-Api-Version` header to pin it — see [Targeting a specific version](/connect/api-versioning/targeting-a-version).
</Note>

## Headers

<ParamField header="X-Nabla-Api-Version" type="string" required placeholder="2026-03-23">
  Pins the API version. Must be `2026-03-23` or later.
</ParamField>

## Body

<ParamField body="provider_email" type="string" required>
  Provider email from your EHR. Must match the email on file when the user already exists.
</ParamField>

<ParamField body="external_provider_id" type="string" required>
  Stable external provider identifier from your EHR. Maximum 256 characters.
</ParamField>

<ParamField body="settings" type="object">
  When present, sets specialty and speech locales. Omit to leave the current values untouched (or to fall back to defaults for a brand-new user).

  <Expandable title="settings">
    <ParamField body="specialty" type="object | null">
      Provider specialty. Set to `null` to clear.

      <Expandable title="specialty">
        <ParamField body="kind" type="enum" required>
          One of the [specialty kinds](#specialty-kinds).
        </ParamField>
        <ParamField body="other_specialty_name" type="string | null">
          Free-form specialty name. Only used when `kind` is `OTHER`.
        </ParamField>
      </Expandable>
    </ParamField>

    <ParamField body="speech_locale" type="enum" required>
      Primary dictation locale. One of the [speech locales](#speech-locales).
    </ParamField>

    <ParamField body="secondary_speech_locale" type="enum | null">
      Optional secondary dictation locale for multilingual providers. One of the [speech locales](#speech-locales).
    </ParamField>
  </Expandable>
</ParamField>

## Behavior

- **Lookup, then merge.** Nabla resolves your organization from the OAuth client used to mint the access token, then looks up an existing user by `external_provider_id`. If found, it returns that user and merges in any `settings` you provided. If not found, it creates the account, links the EHR practitioner, and returns the new user.
- **`settings` omission preserves values.** When you leave `settings` out of the request, the current specialty and locales are left untouched. For a brand-new user, defaults apply until you send `settings` or the provider completes in-app onboarding.
- **Idempotent.** Replaying the same payload is safe — re-calling with the same `external_provider_id` returns the same user with `settings` merged.

### 409 Conflict

The request fails with `409 Conflict` when `external_provider_id` and `provider_email` would point at two different existing users — for example, the `external_provider_id` is already linked to a different email on file. Treat this as a wiring bug, not a retryable error: check that `external_provider_id` is stable per provider and that `provider_email` matches the one on file. See [Provision users](/connect/guides/provision-users#handling-409-conflict).

```bash cURL
curl -X POST https://us.api.nabla.com/v1/connect/server/users \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "X-Nabla-Api-Version: 2026-03-23" \
  -H "Content-Type: application/json" \
  -d '{
    "provider_email": "dr.smith@example-clinic.com",
    "external_provider_id": "prov_8421",
    "settings": {
      "specialty": { "kind": "FAMILY_MEDICINE", "other_specialty_name": null },
      "speech_locale": "ENGLISH_US",
      "secondary_speech_locale": "SPANISH_MX"
    }
  }'
```
